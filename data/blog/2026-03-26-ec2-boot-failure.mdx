## 배경

개발 서버는 비용 절감을 위해 평일 09:00 자동 시작, 18:00 자동 종료 스케줄로 운영 중이다.
매일 재부팅되는 환경인데, 어느 날 아침 특정 서버가 올라오지 않아 확인해보니
부팅 자체가 실패한 상태였다.

---

## 증상

- 2026-03-26 09:02 스케줄 자동 시작 시 부팅 실패
- AWS 콘솔에서 인스턴스 상태 검사 실패
- SSH 접속 불가
- EC2 직렬 콘솔 접속 불가 (이유는 아래에서 설명)
- 인스턴스 스크린샷에서 `Welcome to GRUB!` 화면에서 멈춰있음

| 항목          | 내용                         |
| ------------- | ---------------------------- |
| 인스턴스 타입 | t2.large                     |
| OS            | Amazon Linux 2023            |
| 하이퍼바이저  | Xen (구형)                   |
| 운영 스케줄   | 평일 09:00 시작 / 18:00 종료 |

## 원인 분석

### 핵심 오류 로그

인스턴스 스크린샷과 시스템 로그에서 아래 메시지가 반복되고 있었다.

```
kernel: Cannot get hvm parameter CONSOLE_EVTCHN (18): -22!
kernel: Cannot get hvm parameter CONSOLE_EVTCHN (18): -22!
```

### 오류 의미

- `CONSOLE_EVTCHN` — Xen 하이퍼바이저와 게스트(VM) 간 콘솔 통신에 사용하는 채널 번호
- `-22` — 리눅스 에러 코드 `EINVAL` (Invalid argument), Xen이 해당 파라미터를 모른다는 의미

### 왜 간헐적으로 발생하는가

t2는 구형 **Xen 하이퍼바이저** 기반 인스턴스다.
부팅 시 커널이 Xen에게 콘솔 채널 번호를 요청하는데, Xen이 이 파라미터를 지원하지 못하면 `-22`를 반환한다.

평소에는 커널이 이 오류를 무시하고 부팅을 계속 진행한다.
하지만 **타이밍 이슈**로 인해 간헐적으로 부팅이 중단되는 경우가 생긴다.
매일 재부팅하는 환경에서는 이 확률이 누적되어 결국 발생했다.

### 부팅 이력

```
Mar 16 ~ Mar 25  매일 09:00 정상 부팅 ✅
Mar 26 09:02     부팅 실패 (1분 만에 중단) ❌
Mar 26 10:04     수동 재부팅 후 정상 복구 ✅
```

---

## 왜 직렬 콘솔로 접근이 안됐나

EC2 직렬 콘솔은 **Nitro 하이퍼바이저 기반 인스턴스에서만 지원**된다.
t2는 구형 Xen 기반이라 직렬 콘솔이 미지원이다.

그래서 부팅 실패 시 내부로 접근할 방법 자체가 없었고,
결국 스크린샷으로 상태만 확인 후 재부팅하는 것 외에 선택지가 없었다.

---

## 해결

### 임시 해결

수동 재부팅으로 정상 복구했다.
재부팅 후 같은 오류 로그가 출력됐지만 이번엔 커널이 무시하고 정상 부팅됐다.

### 근본 해결 — t2 → t3 인스턴스 타입 변경

재발 방지를 위해 인스턴스 타입 변경을 권고했다.

```
1. 인스턴스 중지 (Stop)
2. 인스턴스 유형 변경 → t3.large 선택
3. 인스턴스 시작
```

| 항목         | t2.large        | t3.large     |
| ------------ | --------------- | ------------ |
| 하이퍼바이저 | Xen (구형)      | Nitro (최신) |
| 직렬 콘솔    | ❌ 미지원       | ✅ 지원      |
| 이 오류 재발 | 가능            | 발생 안 함   |
| 비용         | 상대적으로 높음 | 약간 저렴    |

t3로 변경하면 Nitro 기반이 되어 `CONSOLE_EVTCHN` 오류 자체가 사라지고,
부팅 실패 시에도 직렬 콘솔로 직접 접속해서 복구가 가능해진다.

## 정리

| 항목      | 내용                                                                   |
| --------- | ---------------------------------------------------------------------- |
| 발생 원인 | t2(Xen) 기반 인스턴스에서 커널-하이퍼바이저 콘솔 채널 협상 타이밍 이슈 |
| 트리거    | 매일 재부팅하는 스케줄 운영 환경                                       |
| 임시 해결 | 수동 재부팅                                                            |
| 근본 해결 | t2 → t3 인스턴스 타입 변경                                             |
| 추가 이점 | Nitro 전환으로 직렬 콘솔 지원 → 이후 부팅 장애 시 복구 용이            |

스케줄 운영처럼 매일 재부팅이 반복되는 환경에서는
구형 Xen 기반 t2 인스턴스는 이런 간헐적 부팅 실패 리스크가 있다.
t3 이상으로 올려두는 것이 안정성 측면에서도, 비용 측면에서도 낫다.
