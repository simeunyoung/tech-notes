## 문제 상황

AWS S3에 퍼블릭 액세스 차단 정책을 적용하고 CloudFront를 통해 이미지를 서빙하는 구조에서,
같은 파일명으로 이미지를 교체 업로드해도 **변경된 이미지가 반영되지 않는 현상**이 발생했다.

원인은 CloudFront가 기존 파일명으로 이미 캐시를 가지고 있어서,
S3에 새 파일을 올려도 캐시가 만료되기 전까지는 계속 이전 이미지를 반환하기 때문이다.

---

## 해결 전략 비교

| 전략                 | 장점                      | 단점                       | 비용            |
| -------------------- | ------------------------- | -------------------------- | --------------- |
| ① Cache Invalidation | 기존 URL 유지             | 수동 작업, 전파 1~5분 소요 | 월 1,000건 무료 |
| ② Cache-Control 헤더 | 자동 만료                 | TTL 내 즉시 반영 불가      | 무료            |
| ③ 쿼리스트링 버전    | 즉시 반영, 구현 쉬움      | 캐시 정책 수정 필요        | 무료            |
| ④ 타임스탬프 파일명  | 즉시 반영, 캐시 충돌 없음 | 파일명 관리 필요           | 무료            |

## 채택 전략 — 타임스탬프 기반 파일명

### 선택 이유

- 기존 CloudFront 캐시 정책 변경 불필요
- Cache Invalidation 비용 및 전파 대기 시간 없음
- 새로운 URL로 요청되므로 CloudFront가 반드시 S3에서 최신 파일을 가져옴
- 서버 코드에서 타임스탬프 생성 및 관리가 단순함

### 파일명 구조

```
# 변경 전
icon/type/category/image.png

# 변경 후
icon/type/category/image_20240301153000.png
```

파일명 뒤에 서버 시간 기반 타임스탬프를 붙이는 단순한 구조다.
서버는 한국 시간(KST, UTC+9) 기준으로 시간을 생성한다.
이미지를 교체할 때마다 타임스탬프가 바뀌므로 URL 자체가 달라진다.

> **참고 — KST 서버 시간 사용 시 유의점**
>
> 서버 시간이 KST 기준이면 파일명에 한국 시간이 그대로 들어간다.
> 파일명만으로 생성 시각을 파악할 수 있어서 운영 편의성은 높지만,
> 서버 타임존 설정이 바뀌거나 다중 서버 환경에서 서버마다 시간이 다를 경우
> 파일명이 충돌하거나 순서가 꼬일 수 있다.
> 서버 타임존이 KST로 고정되어 있다면 실운영에서 큰 문제는 없다.

### 동작 원리

```
1. 새 이미지 준비
   └─ image.png (신규)

2. 서버 시간(KST) 기반 타임스탬프 생성 및 파일명 조합
   └─ timestamp = 20240301153000  (예: 2024-03-01 15:30:00 KST)
   └─ filename  = image_20240301153000.png

3. S3 업로드
   └─ s3://bucket/icon/type/category/image_20240301153000.png

4. DB 또는 설정에 새 파일명 업데이트
   └─ image_url = '.../image_20240301153000.png'

5. 서버 응답 → 클라이언트가 새 URL로 요청
   └─ CloudFront 캐시 미스 → S3에서 최신 이미지 반환
```

---

## 최종 아키텍처

```
클라이언트
   │  GET .../icon/type/category/image_20240301153000.png
   ▼
CloudFront (CachingOptimized)
   │  처음 요청 → 캐시 없음 (MISS)
   ▼
S3 (퍼블릭 액세스 차단)
   │  OAC로 CloudFront만 접근 허용
   │  파일 반환 → CloudFront 캐시 저장 (1년)
   ▼
이후 동일 URL 요청 → CloudFront 캐시 히트

이미지 교체 시:
   타임스탬프 갱신 → 새 URL → 자동으로 새 캐시 생성
```

S3는 퍼블릭 액세스를 차단하고 OAC(Origin Access Control)로 CloudFront만 접근을 허용한다.
CloudFront는 처음 요청 시 S3에서 파일을 가져와 캐시에 저장하고, 이후 동일 URL은 캐시에서 바로 응답한다.

이미지를 교체할 때는 타임스탬프를 갱신해 새 URL을 만들면 자동으로 새 캐시가 생성되므로
Invalidation 없이 즉시 반영된다.

## 운영 시 고려사항

이 전략은 구조가 단순한 대신 운영하면서 챙겨야 할 부분이 있다.

### S3 파일 누적

이미지를 교체할 때마다 새 파일이 추가되고 기존 파일은 그대로 남는다.
교체가 잦은 이미지일수록 같은 경로에 파일이 쌓여 스토리지 비용이 올라간다.

S3 Lifecycle 정책으로 일정 기간이 지난 파일을 자동 삭제하는 것을 권장한다.
업로드 시점에 이전 파일을 직접 삭제하는 방법도 있지만,
현재 사용 중인 파일을 실수로 지우는 케이스가 생길 수 있어서 Lifecycle 정책이 더 안전하다.

### 이전 URL 캐시 잔존

이미지를 교체해도 이전 타임스탬프 URL을 캐싱하고 있는 클라이언트는 계속 이전 이미지를 본다.
서버가 항상 최신 URL을 응답으로 내려주는 구조라면 자연스럽게 해소되지만,
앱처럼 URL을 로컬에 저장해두는 클라이언트가 있다면 별도 갱신 처리가 필요할 수 있다.

### CloudFront 고아 캐시 누적

타임스탬프가 바뀌면 이전 URL의 CloudFront 캐시는 TTL이 만료될 때까지 그대로 남는다.
TTL이 1년이면 사용되지 않는 캐시가 오래 유지된다.
비용에 직접적인 영향은 없지만 신경 쓰인다면 교체 시점에 이전 URL만 Invalidation 해주면 된다.

### KST 서버 시간 사용 시 유의점

현재 서버 시간(KST) 기반으로 타임스탬프를 생성하고 있어서,
서버 타임존 설정이 바뀌거나 다중 서버 환경에서 서버마다 시간이 다를 경우
파일명이 충돌하거나 순서가 꼬일 수 있다.
서버 타임존이 KST로 고정된 단일 서버 환경이라면 실운영에서 큰 문제는 없다.

| 사이드이펙트         | 해결 방법                                    |
| -------------------- | -------------------------------------------- |
| S3 파일 누적         | S3 Lifecycle 정책으로 오래된 파일 자동 삭제  |
| 이전 URL 캐시 잔존   | 서버에서 항상 최신 URL 내려주는 구조 유지    |
| CloudFront 고아 캐시 | 필요 시 이전 URL만 선택적 Invalidation       |
| KST 시간 의존        | 서버 타임존 KST 고정 유지, 다중 서버 시 주의 |

## 정리

| 항목 | 내용                                                      |
| ---- | --------------------------------------------------------- |
| 문제 | 동일 파일명 교체 시 CloudFront 캐시로 인해 이미지 미반영  |
| 해결 | 서버 시간(KST) 기반 타임스탬프 파일명으로 URL 자체를 변경 |
| 장점 | Invalidation 불필요, 즉시 반영, 추가 비용 없음            |
| 단점 | 파일명과 URL을 DB 또는 설정에서 관리해야 함, S3 파일 누적 |

Cache Invalidation도 간단한 해결책이지만 전파 시간이 있고 건수가 쌓이면 비용이 발생한다.
타임스탬프 파일명 전략은 구조를 바꾸지 않고 파일명 규칙 하나로 문제를 해결할 수 있어서
운영 부담이 적은 방식이다. 다만 S3 파일 누적은 초기에 Lifecycle 정책을 걸어두는 것이 좋다.
