## 배경

IoT 플랫폼에서 OTA(Over-The-Air) 펌웨어 업데이트 기능을 운영 중이었다.
흐름은 아래와 같다.

```
1. 관리자가 IoT Core에 OTA Job 생성
2. 대상 디바이스가 QUEUED 상태로 DB에 등록
3. 앱에서 OTA 대상 디바이스 목록과 상태 표시
4. 앱에서 OTA 시작
5. 디바이스가 업데이트를 진행하면서 상태 변경
   QUEUED → IN_PROGRESS → SUCCEEDED
6. 앱에서 진행 현황 조회
```

앱은 서버 DB에서 상태를 조회해 현황을 보여주는 구조다.

---

## 문제 증상

두 가지 시점에서 지연이 발생했다.

**1. OTA 작업 등록 시**
관리자가 IoT Core에 Job을 생성해도 앱에 OTA 표시가 바로 뜨지 않았다.
폴링 주기가 될 때까지 앱에는 아무것도 표시되지 않았다.

**2. OTA 완료 시**
디바이스가 업데이트를 완료해도 앱의 OTA 표시가 즉시 사라지지 않았다.
완료 후에도 몇 초간 진행 중 상태가 유지됐다.

---

## 기존 구조 분석

기존 Lambda의 동작 방식은 아래와 같았다.

```
EventBridge (5분 주기)
    ↓ 트리거
Lambda: 상태 처리 함수
    ↓
IoT Core API: list_job_executions_for_job (폴링)
    ↓ 페이지네이션으로 전체 디바이스 조회
RDS: INSERT or UPDATE
```

동작 흐름은 아래 순서였다.

1. 진행 중인 Job 전체를 DB에서 조회
2. Job마다 IoT Core API를 직접 폴링해서 디바이스 목록 조회
3. 전체 디바이스를 순회하며 DB에 상태 등록

이 구조의 문제점은 세 가지였다.

**1. 폴링 주기만큼 지연**
EventBridge가 주기마다 Lambda를 깨우는 구조라
디바이스가 상태를 바꿔도 다음 폴링 주기까지 DB가 갱신되지 않는다.
폴링 간격이 길수록 지연도 그만큼 커진다.

**2. 불필요한 Lambda 반복 실행**
진행 중인 Job이 없어도 폴링 주기마다 무조건 Lambda가 실행되고,
Job이 있으면 `list_job_executions_for_job`으로 전체 디바이스를 페이지네이션해서 긁어온다.
실제 변화가 없는데도 매번 전체를 조회하는 낭비가 발생했다.

**3. 상태 전이 가드 없음**
`SUCCEEDED`가 된 디바이스도 다음 폴링 주기에 `IN_PROGRESS`로 다시 덮어쓰일 수 있는 구조였다.

---

## 원인

근본 원인은 IoT Core의 상태 변화를 **당겨오는(pull) 방식**으로 설계된 것이다.
디바이스가 상태를 바꾸는 시점과 서버가 그걸 인지하는 시점 사이에
폴링 주기만큼의 gap이 필연적으로 존재한다.

AWS IoT Core는 Job Execution 상태가 바뀔 때마다
`$aws/events/jobExecution/{jobId}/{status}` 토픽으로 MQTT 이벤트를 발행한다.
이걸 이용하면 폴링 없이 상태 변화를 즉시 받을 수 있다.

> **주의:** 이 토픽은 AWS IoT Core Settings에서 Job Execution 이벤트를 명시적으로 활성화해야 메시지가 발행된다.
> 이 설정이 꺼져 있으면 OTA가 성공해도 해당 토픽으로 아무 메시지도 오지 않는다.

## 해결 — 이벤트 드리븐으로 전환

```
IoT Core: 디바이스 상태 변경
    ↓ MQTT 이벤트 발행 ($aws/events/jobExecution/+/+)
IoT Rule (SQL: topic 파싱, status 필터링)
    ↓ 즉시 트리거
Lambda: 상태 처리 함수
    ↓
RDS: UPSERT
```

### IoT Rule SQL

IoT Rule SQL로 필요한 필드를 파싱해서 Lambda에 전달한다.

```sql
SELECT
  topic(4) AS job_id,
  topic(5) AS operation,
  thingArn  AS thing_arn,
  status,
  timestamp() AS received_ts,
  timestamp   AS event_ts
FROM '$aws/events/jobExecution/+/+'
WHERE status <> 'REMOVED'
```

### 상태 전이 가드

상태 역전 문제를 방지하기 위해 상태 전이 가드를 추가했다.
아래 규칙으로 상태 업데이트 여부를 판단한다.

- 기존 상태가 없으면 → 업데이트
- 기존 상태가 `SUCCEEDED`이면 → 업데이트 차단 (최종 상태)
- 기존 상태가 `FAILED`이면 → `SUCCEEDED`로의 변경만 허용 (재시도 성공 케이스)
- 그 외 → 업데이트

## 결과

| 항목        | 변경 전                    | 변경 후                    |
| ----------- | -------------------------- | -------------------------- |
| 상태 반영   | 폴링 주기만큼 지연         | 즉시 반영                  |
| Lambda 실행 | 폴링 주기마다 무조건 실행  | 이벤트 발생 시에만 실행    |
| API 호출    | 전체 디바이스 페이지네이션 | 변경된 디바이스 1건만 처리 |
| 상태 역전   | 발생 가능                  | 상태 전이 가드로 차단      |

폴링 구조를 제거하고 이벤트 드리븐으로 전환해서
디바이스 상태가 변경되는 즉시 DB에 반영된다.
실제 이벤트가 있을 때만 Lambda가 실행되어 불필요한 API 호출과 실행 비용도 제거됐다.
